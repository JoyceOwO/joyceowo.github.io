## 引言

你可能寫好一個 .NET 工具或共用元件，想把它變成 NuGet 套件，但流程常卡在「怎麼推？怎麼測？」。

這篇直接帶你走完 **NuGet 套件發布與本地測試流程**，讓你不用猜、不用踩坑。

---

## 🎯 目標說明

這篇要解決的事情很單純：

- ✔ 將 `.nupkg` 套件推送到 NuGet Server
- ✔ 在本地快速測試套件（不必真的發佈）
---

## 核心步驟

### 步驟一：推送 NuGet 套件到伺服器

```bash
# 將套件推送到指定 NuGet Server
dotnet nuget push \\
  -s <https://xxx.yyy.zzz/v3/index.json> \\   # NuGet Server URL
  -k KEYXXX1234 \\                          # API Key
  AppLogger.1.0.0.nupkg                   # 套件檔案
```

**說明：**

- `s`：指定 NuGet Server（可為私有或官方）
- `k`：驗證用 API Key
- `.nupkg`：你的套件檔
**為什麼這樣做？**

- NuGet Server 是套件分發中心
- push 後，其他專案才能透過版本引用
**適用情境：**

- 公司內部共用套件
- 公開套件（[nuget.org](http://nuget.org/)）
**限制：**

- API Key 錯誤會直接失敗
- 套件版本不可重複（需遞增）
---

### 步驟二：本地測試套件（不用真的發佈）

### 方法一：直接加入專案參考（最快）

1. 方案右鍵 → 加入現有專案
1. 選擇你的「套件專案」
1. 在使用端專案：
  - 右鍵 → 相依性
  - 新增專案參考
  - 選擇剛加入的套件專案
**優點：**

- 不需要打包
- 即時修改即時測試
**為什麼這樣做？**

- 避免每次改一行就重新打包 `.nupkg`
---

### 方法二：使用本地 NuGet Source（模擬正式環境）

```bash
# 將 nupkg 放到本地資料夾，例如：
C:\\local-nuget
```

然後在 NuGet 設定中加入來源：

```bash
dotnet nuget add source "C:\\local-nuget" -n Local
```

**說明：**

- 模擬「真正安裝套件」的流程
- 適合驗證版本、依賴
---

## ⚠️ 常見問題 / 踩坑

### 1. 套件推不上去

**原因：**

- API Key 錯誤
- URL 錯誤
- 版本已存在
**解法：**

- 確認 Key 與 Server
- 更新版本號（如 1.0.1）
---

### 2. 引用不到最新版本

**原因：**

- NuGet 快取問題
**解法：**

```bash
dotnet nuget locals all --clear
```

---

### 3. 本地測試與正式行為不同

**原因：**

- 專案參考 ≠ NuGet 套件引用
**解法：**

- 用「本地 NuGet Source」再測一次
