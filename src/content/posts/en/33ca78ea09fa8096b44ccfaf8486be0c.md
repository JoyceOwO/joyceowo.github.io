## Conclusion

Overwriting LinkServer logins with 'fake account mapping' to make specified users always fail to connect is the simplest and most controllable blocking method.

## Use Cases

- When you want to restrict 'specific accounts' from using LinkServer
- When you don't want to affect other users (maintaining public availability)
- When you cannot precisely control LinkServer with DENY
- When permissions are already chaotic and you need a quick fix

## Process Steps

### 1. Confirm Current LinkServer Login Settings

- Use `sp_helplinkedsrvlogin` to view mappings
- You can see who maps to which remote account
- Determine if there is `NULL` (meaning shared by everyone)
```sql
EXEC sp_helplinkedsrvlogin;
```

### 2. Remove Existing Mappings for the Target User

- For all LinkServers
- Delete the login mapping for that User
- To prevent interference from old settings
```sql
EXEC sp_droplinkedsrvlogin
    @rmtsrvname = N'LinkServerName',
    @locallogin = N'UserA';
```

### 3. Create a 'Fake Account' to Force Login Failure

- Core approach: Specify a non-existent account
- When the User calls LinkServer → login will definitely fail
- Equivalent to a 'soft DENY'
```sql
EXEC sp_addlinkedsrvlogin
    @rmtsrvname = N'LinkServerName',
    @useself = N'False',
    @locallogin = N'UserA',
    @rmtuser = N'FakeDenyUser',
    @rmtpassword = N'WrongPassword';
```

## Additional Notes

- There is no `DENY USE LINKSERVER` permission; control can only be done via mapping
- `@locallogin = NULL` means shared by everyone; special attention is needed for the overwrite order
- This method does not affect other users (only targets specified accounts)

## Wrap-up

LinkServer lacks permission control? Then use the trick of 'making them unable to connect forever' – it's clean and decisive.
Practical advice: Design permissions by separating accounts from the start, otherwise, you'll only be patching holes later.
