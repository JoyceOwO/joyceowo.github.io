## Conclusion
- Outdated SMO versions cannot parse newer SQL Server (v16). Updating SSMS resolves this.
## Where This Applies
- Using SMO to retrieve Table / View Scripts
- Automatically generating SQL Schema
- Environments using older SSMS / older SDKs
- Encountering "version 16" / "version not supported" errors
## Steps

### 1. Identify the Error Source (SMO Version Not Supported)
- `invalid version 16` appears when executing a command to retrieve Create Table / View syntax.
- This indicates that SMO cannot recognize SQL Server 2022 (v16).
- Commonly seen in:
  - Older SSMS versions
  - Older NuGet packages: Microsoft.SqlServer.Smo
![image](../../../assets/notion-images/23ba78ea09fa8185bbdedd283180ed59/23ba78ea09fa8185bbdedd283180ed59-7e6d5d8af3d2.png)

### 2. Update SSMS (Quickest Solution)
- Download the latest SSMS from Microsoft's official website.
- Installing it will automatically update related SMO components.
- Why:
  - SMO versions are tied to SSMS.
  - Newer SSMS versions support SQL Server 2022+.
### 3. (Programmatic) Update NuGet Packages (If Using SMO)
- Update the SMO package version.
```powershell
# 更新 SMO 套件
dotnet add package Microsoft.SqlServer.SqlManagementObjects
```

- Ensure the version supports SQL Server 2022 or newer.
## Additional Notes
- SSMS and SQL Server versions **do not need to match, but SSMS cannot be too old**.
- SMO is very sensitive to versions; it's a client-side issue, not a DB issue.
- If your CI/CD pipeline uses SMO, you must also upgrade the package.
## Wrap-up
- If you see "version 16," don't doubt yourself; nine times out of ten, your tools are too old. Just update them.
