## 引言

如果你在使用 EF Core 時，遇過「同一段查詢，有時 1 秒，有時 timeout」的詭異現象，那你大概已經踩進 **Parameter Sniffing** 的坑裡了。

這不是你寫錯，也不是 DB 壞掉，而是 SQL Server 在「幫你猜最佳執行計畫」，但有時候猜錯，代價就是效能崩壞。

這篇會帶你用一個**優雅又可控的方式**：

👉 在 EF Core 中動態加上 `OPTION(RECOMPILE)`

讓每次查詢都重新編譯，避免錯誤的執行計畫拖垮效能。

---

## 🎯 目標說明

這篇要幫你做到：

- 在 EF Core Query 層級 **精準控制 RECOMPILE**
- 不影響其他查詢（避免全域副作用）
- 保持 LINQ 使用習慣
---

## 核心實作

### 步驟一：註冊 Interceptor（DbContext 設定）

```c#
builder.UseSqlServer(connectionString)
    .UseRecompileExtensions(); // 註冊自訂 Interceptor
```

### 為什麼這樣做？

- EF Core 提供 `DbCommandInterceptor`
- 可以在 SQL 發送前「偷改 SQL」
- 是最乾淨、不侵入 LINQ 的方式
---

### 步驟二：在查詢上加 `.WithRecompile()`

```c#
var name = "SomeName";

var result = context.SomeItems
    .Where(x => x.Name == name)
    .WithRecompile() // 標記這個查詢要 RECOMPILE
    .ToList();
```

### 實際產生 SQL

```sql
SELECT [s].[Id], [s].[Name]
FROM [SomeItems] AS [s]
WHERE [s].[Name] = @__name_0
OPTION(RECOMPILE)
```

### 為什麼這樣做？

- 只針對「特定查詢」開 RECOMPILE
- 不會影響整個系統
- 適合資料分布極端不平均的場景
---

### 步驟三：實作 Extension + Interceptor

```c#
public static class RecompileExtensions
{
    private const string RecompileTag = "recompile_query_tag";
    private const string RecompileComment = "-- " + RecompileTag + "\\r\\n";

    // 註冊 Interceptor
    public static DbContextOptionsBuilder UseRecompileExtensions(this DbContextOptionsBuilder builder)
    {
        return builder.AddInterceptors(RecompileInterceptor.Instance);
    }

    // 在 LINQ 上標記
    public static IQueryable<T> WithRecompile<T>(this IQueryable<T> query)
    {
        return query.TagWith(RecompileTag); // 利用 TagWith 插入 SQL comment
    }

    private class RecompileInterceptor : DbCommandInterceptor
    {
        public static RecompileInterceptor Instance = new();

        public override ValueTask<InterceptionResult<DbDataReader>> ReaderExecutingAsync(
            DbCommand command,
            CommandEventData eventData,
            InterceptionResult<DbDataReader> result,
            CancellationToken cancellationToken = new())
        {
            CorrectCommand(command);
            return base.ReaderExecutingAsync(command, eventData, result, cancellationToken);
        }

        public override InterceptionResult<DbDataReader> ReaderExecuting(
            DbCommand command,
            CommandEventData eventData,
            InterceptionResult<DbDataReader> result)
        {
            CorrectCommand(command);
            return base.ReaderExecuting(command, eventData, result);
        }

        private static void CorrectCommand(DbCommand command)
        {
            var newQuery = command.CommandText.Replace(RecompileComment, "");

            // 如果有標記，代表需要加 RECOMPILE
            if (!ReferenceEquals(newQuery, command.CommandText))
            {
                // 清掉前面的換行
                if (newQuery.StartsWith("\\r\\n"))
                    newQuery = newQuery.Substring(2);

                newQuery += "\\r\\nOPTION(RECOMPILE)";
                command.CommandText = newQuery;
            }
        }
    }
}
```

### 背後原理

- `TagWith()` 會在 SQL 前加上 comment
- Interceptor 偵測 comment → 判斷是否需要 RECOMPILE
- 再動態改寫 SQL
👉 有點像在 SQL 上「偷偷貼標籤再加工」

## ⚠️ 常見問題 / 踩坑

### 1. RECOMPILE 會變快嗎？

不一定。

| 情境 | 結果 |
| --- | --- |
| Parameter Sniffing 嚴重 | 👍 明顯改善 |
| 查詢本來就穩定 | 👎 可能變慢（多編譯成本） |

---

### 2. 可以全域開嗎？

```sql
ALTER DATABASE SCOPED CONFIGURATION
SET PARAMETER_SNIFFING = OFF;
```

👉 可以，但不建議

原因：

- 全部查詢都失去 plan cache
- CPU 會上升
- 很難回頭調整
👉 建議：**只針對問題查詢使用**

---

### 3. Interceptor 會影響其他查詢嗎？

不會，只會影響有 `.WithRecompile()` 的查詢。

---

### 4. 可以用在哪些場景？

適用：

- 條件差異極大的查詢（熱門 vs 冷門資料）
- IN / JSON 查詢（像 OPENJSON）
- 報表查詢
不適用：

- 高頻短查詢（會增加編譯成本）
- 穩定 OLTP 查詢
---

## 總結

這篇你做了三件事：

- 用 `TagWith` + Interceptor，優雅控制 SQL 行為
- 在 EF Core 中動態加上 `OPTION(RECOMPILE)`
- 精準解決 Parameter Sniffing 問題，而不是亂開全域設定
記住一句話：

> RECOMPILE 不是萬靈丹，但用對地方，是救命丹。

有些坑，你踩過才會懂。

但如果能少踩一次，那這篇就值了 🙂
